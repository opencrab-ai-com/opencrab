# OpenCrab 官网首页视觉与版式方案 v1

日期：2026-03-22  
角色：审美设计师 / Design  
对应输入：[`productA-homepage-v1.md`](/Users/sky/Desktop/OpenCrabTeams/Case1/productA-homepage-v1.md)

## 1. 总判断

OpenCrab 官网首页不该做成“泛 AI 工具官网”，而应该做成一种更克制的产品型首页：**像一个安静、可信、可操作的本地 AI 工作台，而不是一个热闹的概念片。**

页面气质建议收敛为这句话：

**Quiet editorial software. Local-first workspace.**

中文可以理解为：

**有一点编辑感，但本质上是产品感；有辨识度，但不靠花哨效果撑场。**

## 2. 整体设计方向

### 2.1 方向关键词

- 本地工作台
- 安静的力量
- 编辑感排版
- 温暖留白
- 产品证据优先
- 克制的品牌感

### 2.2 可借鉴的海外 AI 官网元素

可以借：

- `OpenAI / ChatGPT` 的首屏克制感：一句核心价值 + 极少按钮 + 大留白
- `Claude` 的温和背景与编辑感标题：让页面更像一个有判断的产品，而不是开发工具文档
- `Cursor` 的产品证据方式：尽早出现真实界面，而不是长时间只讲理念
- `OpenClaw` 的 frontstage / backstage 气质：适合借到 Team Runtime 预告区，用来表达“前台协作 + 后台执行”

不要照搬：

- 不要做纯黑科技风
- 不要做大面积紫蓝渐变
- 不要把首屏做成抽象品牌片
- 不要把导航、入口、二级链接堆成 SaaS 营销站

### 2.3 建议主视觉主轴

首页主轴建议定为：

**沙色纸感背景 + 深墨文字 + 克制珊瑚橙品牌点缀 + 真实产品截图作为第一证据。**

这样能同时接住三件事：

- 和现有产品内部偏暖白、细描边、轻阴影的界面语言连续
- 保留 OpenCrab 自己的品牌橙，不会被做成“另一个米白色 AI 官网”
- 高级感来自比例、留白、字体和节奏，不靠特效堆出来

## 3. 首页叙事与版式节奏

### 页面目标

用户滚到页尾前，应该完成三件判断：

1. OpenCrab 不是单纯聊天页，而是能在同一工作空间里继续做事
2. 它的差异化来自本地优先、任务、渠道、团队协作方向
3. 现在就可以先安装体验，而不是等一个未来概念

### 建议版块顺序

1. 顶部导航
2. Hero 首屏
3. 价值确认带：不是另一个 AI 聊天页
4. 四种工作方式
5. 三类典型场景
6. 本地优先说明
7. Team Runtime 预告
8. 产品截图带
9. 安装门槛与最终 CTA
10. FAQ / 页脚

### 节奏原则

- 首屏重判断，第二屏重差异，第三屏才展开能力
- 文字密度和视觉密度交替出现，避免连续三段重文案
- 每滚动一屏只让用户接受一个新判断，不并列讲三件大事
- CTA 全页保持一致，以“安装 / GitHub”作为主次关系，不要后面反复换口径

## 4. 首页结构说明

### 4.1 顶部导航

结构：

- 左：OpenCrab 标志
- 中：`能力` `场景` `Team Runtime` `GitHub`
- 右：主 CTA `开始安装`

要求：

- 导航高度 72px
- 透明起始，滚动后变为半透明暖白浮层
- 不做复杂下拉
- GitHub 作为外链次入口即可，不要抢主按钮

### 4.2 Hero 首屏

#### 首屏任务

- 让用户 3 秒内知道 OpenCrab 是什么
- 让用户立刻看到“真实产品”
- 把“本地优先的 AI 工作台”打进记忆点

#### 建议结构

桌面端用 `5 : 7` 左右分栏：

- 左侧文案区
- 右侧产品舞台区

移动端改为：

- 文案在上
- 主截图在下
- 浮动小卡片减少到 1 到 2 张

#### 左侧信息层级

顺序：

1. 小标签：`Local-first AI Workspace`
2. H1：**把聊天、执行、任务和渠道，收进同一个本地优先的 AI 工作台。**
3. 副标题：一句话解释聊天主入口、文件、浏览器、定时任务、Telegram / 飞书、Team Runtime 的关系
4. CTA 行：
   - 主 CTA：`开始安装`
   - 次 CTA：`查看 GitHub`
5. 辅助证明短标签：
   - `macOS`
   - `Codex`
   - `Local runtime`
   - `中文优先`

#### 右侧产品舞台区

建议不是单张大图，而是一个主界面 + 两个辅助浮层：

- 主图：聊天首页截图，放最大
- 浮层 1：定时任务卡片，强调“按计划执行”
- 浮层 2：渠道卡片，强调“同一工作空间”

布局建议：

- 主图占 80% 视觉权重
- 两张浮层错位叠放，不要平均分布
- 浮层只显示关键信息，不显示太多 UI 细节

#### 首屏 CTA 摆放

- 主按钮放在文案区正文下方，不放右图里
- 右图里不要再出现第二套按钮
- 如果要加第三个低权重动作，只能是 `查看安装要求`

### 4.3 价值确认带：不是另一个 AI 聊天页

形式：

- 2 x 2 卡片网格
- 每张卡一个短标题 + 一句解释

建议四张卡：

- 聊天是入口，不是终点
- 定时任务持续推进
- Telegram / 飞书回到同一上下文
- 本地优先，更贴近你的真实工作空间

版式重点：

- 这是首屏后的第一段“判断强化区”
- 卡片不要做太彩
- 标题比正文更有重量，正文最多两行

### 4.4 四种工作方式

形式：

- 四张大卡，桌面端 2 x 2，移动端纵向堆叠
- 每张卡都有小标题、1 句描述、1 个局部截图

四张卡建议固定为：

1. 对话处理工作
2. 文件与浏览器辅助执行
3. 定时任务持续推进
4. 渠道接入真实消息入口

节奏要求：

- 这段是“机制解释区”
- 比上一段更具体，但仍然不进入技术术语
- 卡片高度统一，截图裁切比例统一

### 4.5 三类典型场景

形式：

- 三列场景条带，偏内容型排版

每个场景内容结构：

- 场景名
- 一句结果导向描述
- 一句“之前 vs 现在”的转折式说明

建议三类场景：

- 个人工作台
- 轻自动化工作流
- 多入口协作

这一段要比能力区更“轻”，让用户从功能阅读切换到使用想象。

### 4.6 本地优先说明

形式建议：

- 一张横向长区块
- 左边是观点，右边是 3 条事实

文案气质：

- 不说绝对安全
- 不说企业级合规
- 只说更可控、更贴近本机工作流、更适合长期使用

右侧三条事实建议：

- 对话、附件、运行时状态默认落在本地
- 浏览器连接和工作流由本机环境承接
- 更适合把 AI 接入自己的真实工作空间

### 4.7 Team Runtime 预告

定位：

- 首页上的未来感亮点
- 但必须是 `Preview / 正在进化`，不能抢当前成熟能力

形式：

- 左侧一段简洁说明
- 右侧一张结构示意图或团队群聊界面图

建议标题：

**从单人工作台，延伸到团队协作运行时**

建议说明方式：

- 前台群聊负责共享目标和结果
- 后台多个角色独立执行
- 项目经理负责编排，而不是所有角色一起刷屏

这段更适合做成“静态结构图 + 少量动效”，不适合做大段文案。

### 4.8 产品截图带

用途：

- 给真实证据
- 补足信任

建议顺序：

- 聊天首页
- 定时任务
- 渠道
- Team 界面或技能页

样式要求：

- 截图必须统一容器边框、圆角、投影、背景
- 每张图上方只给一个短标签，不写长说明
- 可以横向滚动，但速度要慢，别做廉价跑马灯

### 4.9 安装门槛与最终 CTA

这一段要承担“筛选预期”的作用。

结构：

- 左：一句明确结论
- 右：安装前提 3 条 + CTA

建议文案：

- 推荐环境：macOS
- 需要可用的 Codex 登录状态
- 适合希望把 AI 接进本地工作流的人

CTA：

- 主 CTA：`开始安装`
- 次 CTA：`查看 GitHub`

## 5. 字体、配色、留白与组件气质

### 5.1 字体建议

推荐方案：

- 标题：`Noto Serif SC` + `Instrument Serif`
- 正文与界面：`Noto Sans SC` + `Instrument Sans`

原因：

- 中文标题需要一点书卷感和编辑感，但不能古风
- 英文字母、数字、按钮和标签需要更干净的现代感
- 这组搭配容易做出“海外 AI 官网的高级感”，同时不牺牲中文可读性

不建议：

- 全站只用系统默认无衬线
- 用过于网红感的几何字体
- 英文高级、中文回落到普通默认字体，导致中英文气质断裂

### 5.2 配色建议

建议保留现有产品的暖白底色逻辑，但把品牌橙用得更节制。

基础色板：

- 页面背景：`#F6F1E8`
- 次级背景：`#FBF8F2`
- 主文字：`#171717`
- 次文字：`#5F5A52`
- 分割线：`rgba(23, 23, 23, 0.08)`
- 深色按钮：`#1D232F`
- 按钮文字：`#F8F4EC`
- 品牌橙：`#F46A4E`
- 深海军蓝：`#24324A`
- 柔和鼠尾草灰：`#A6B2AA`

使用规则：

- 橙色只出现在 Logo、主按钮 hover、小面积状态点和局部强调
- 深海军蓝用于示意线、图标描边、Team Runtime 结构图
- 不要让页面变成橙色官网，橙色只是签名，不是底色

### 5.3 留白和网格

建议：

- 内容最大宽度：`1200px`
- Hero 最大宽度可放宽到 `1280px`
- 桌面端 section 上下留白：`120px - 144px`
- 移动端 section 上下留白：`72px - 88px`
- 卡片内边距：`28px - 32px`
- 主圆角：`24px`
- 小圆角：`14px`

这页的高级感主要靠：

- 大留白
- 稳定网格
- 不拥挤的组件关系
- 统一的圆角和细边框

### 5.4 组件气质

按钮：

- 主按钮实心深色，次按钮白底描边
- 高度 48px
- 文案尽量短，不用图标堆信息

卡片：

- 白底或微暖白底
- 细边框 + 极轻阴影
- 不要彩色大底卡片

标签：

- 使用小写或短词组
- 更像产品状态，不像营销徽章

动效：

- 只在首屏入场、截图轻位移、区块 reveal 使用
- 节奏慢，位移小
- 不做 3D 漂浮秀

## 6. 生图 / 生视频使用判断

### 结论

**适合加，但只能作为氛围层，不是主叙事层。**

### 6.1 生图适合放哪里

- Hero 主图后方，做一层非常淡的抽象纹理
- Team Runtime 预告区，做结构图背后的低对比氛围底
- 页尾 CTA 区，做极轻的海岸纹理收束气氛

适合作用：

- 建立“海岸实验室 / 本地工作台”的品牌气候
- 让大留白不显空

不适合作用：

- 代替产品截图
- 独立撑起首屏
- 生成人像、机器人、赛博城市

### 6.2 生视频适合放哪里

建议只放一段，位置二选一：

1. Hero 背后的低对比静音循环视频
2. Team Runtime 预告区的局部氛围视频

视频要求：

- 时长 6 到 10 秒
- 无声
- 慢速循环
- 不抢文案
- 首帧单独可当静态海报

建议方向：

- 海潮纹理
- 沙面风纹
- 玻璃与矿物缓慢折光
- 低饱和、低速度、低噪声

### 7.1 开发实现优先级

第一优先级：

- 先把版式、留白、截图容器和 CTA 关系做对
- 先不上复杂视频，先用静态图撑起完成度

第二优先级：

- 再补轻动效
- 再决定 Hero 是否上氛围视频

第三优先级：

- 最后才做滚动细节、视差、截图轨道微动效

### 7.2 工程注意点

- 产品截图尽量用真实页面，不要重新画假 UI
- 截图需要统一裁切比例，优先 `16:10` 或 `4:3`
- 所有 section 标题宽度要控制，不要拉成一整行墙
- 移动端首屏 CTA 必须在首屏内完整出现
- 若字体加载成本高，先保证正文稳定，再补标题字体

## 8. 当前缺口

这版已经足够指导首页实现，但还有三项素材缺口会影响最后质感：

- 一组统一风格的官网截图素材，最好重新按官网比例截一遍
- Hero 背景所需的生图或短视频素材
- 最终字体授权与加载方案确认

如果后续进入实现阶段，我建议先由开发按这份结构搭出页面骨架，再做一次基于真实页面的审美收口。
