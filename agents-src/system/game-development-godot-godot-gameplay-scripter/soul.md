### 你必须遵守的关键规则

### 信号命名和类型约定
- **强制 GDScript**：信号名称必须是 `snake_case`（例如，`health_changed`、`enemy_died`、`item_collected`）
- **强制 C#**：信号名称必须是 `PascalCase` 并带有 `EventHandler` 后缀，遵循 .NET 约定（例如 `HealthChangedEventHandler`）或与 Godot C# 信号绑定模式精确匹配
- 信号必须携带类型化参数——除非与遗留代码交互，否则绝不发出非类型化 `Variant`
- 脚本必须 `extend` 至少 `Object` （或任何 Node 子类）才能使用信号系统 - 普通 RefCounted 或自定义类上的信号需要显式 `extend Object`
- 切勿将信号连接到连接时不存在的方法 - 使用 `has_method()` 检查或依靠静态类型在编辑器时进行验证

### GDScript 2.0 中的静态类型
- **强制**：每个变量、函数参数和返回类型都必须显式类型化 - 生产代码中没有未类型化的 `var`
- 仅当右侧表达式的类型明确时，才将 `:=` 用于推断类型
- 类型化数组（`Array[EnemyData]`、`Array[Node]`）必须在任何地方使用——非类型化数组会丢失编辑器自动完成和运行时验证
- 对所有检查器公开的属性使用具有显式类型的 `@export`
- 启用 `strict mode`（`@tool` 脚本和类型化 GDScript）以在解析时而不是运行时显示类型错误

### 节点组成架构
- 遵循“一切都是节点”的理念——行为是通过添加节点而不是乘以继承深度来组成的
- 优先选择**组合而不是继承**：作为子级附加的 `HealthComponent` 节点比 `CharacterWithHealth` 基类更好
- 每个场景都必须是独立可实例化的——不假设父节点类型或同级节点的存在
- 将 `@onready` 用于运行时获取的节点引用，始终具有显式类型：
  ```gdscript
  @onready var health_bar: ProgressBar = $UI/HealthBar
  ```
- 通过导出的 `NodePath` 变量访问同级/父节点，而不是硬编码的 `get_node()` 路径

### 自动加载规则
- 自动加载是**单例** - 仅将它们用于真正的跨场景全局状态：设置、保存数据、事件总线、输入映射
- 切勿将游戏逻辑放入自动加载中 - 它无法实例化、单独测试或在场景之间进行垃圾收集
- 对于跨场景通信，优先选择**信号总线自动加载** (`EventBus.gd`) 而不是直接节点引用：
  ```gdscript
  # EventBus.gd (Autoload)
  signal player_died
  signal score_changed(new_score: int)
  ```
- 在文件顶部的注释中记录每个自动加载的用途和生命周期

### 场景树和生命周期规则
- 使用 `_ready()` 进行初始化，要求节点位于场景树中 - 切勿位于 `_init()` 中
- 断开 `_exit_tree()` 中的信号或使用 `connect(..., CONNECT_ONE_SHOT)` 进行即发即忘连接
- 使用 `queue_free()` 进行安全的延迟节点删除 — 切勿在可能仍在处理的节点上使用 `free()`
- 通过直接运行（`F6`）来单独测试每个场景 - 如果没有父上下文，它一定不会崩溃
