# opencrab V1 初版路由结构

## 1. 目标

这份文档用于定义 `opencrab` V1 Web 端的页面路由结构。

当前重点回答：
- 哪些页面需要独立路由
- 哪些状态更适合放在 URL 中
- 哪些交互只需要本地 UI 状态，不需要单独路由

当前不展开：
- 具体前端框架路由实现
- 服务端渲染策略
- 权限守卫细节
- 页面级代码拆分细节

## 2. 路由设计原则

- 路由优先表达一级页面和可分享的详情状态
- 左侧一级入口尽量一一映射到主路由
- 同一页面骨架下的切换优先通过参数态完成，而不是拆太多独立路由
- 弹窗、下拉、输入框草稿、选择器等状态不进路由
- V1 先保持路由简单，避免过早设计复杂嵌套路由

## 3. 顶层路由

V1 建议先收敛为这些顶层路由：

- `/`
- `/conversations`
- `/channels`
- `/tasks`
- `/skills`
- `/settings`

说明：
- `/` 作为首页入口，默认进入聊天主界面
- `/conversations` 对应 `对话`
- `/channels` 对应 `Channels`
- `/tasks` 对应 `任务`
- `/skills` 对应 `Skills`
- `/settings` 对应左下角低频设置页

## 4. 首页与对话路由

### 4.1 `/`

用于承载首页空态。

页面表现：
- 左侧栏正常显示
- 主区显示品牌标题、欢迎文案和输入框
- 不展示 dashboard 卡片和快捷入口

V1 说明：
- 如果用户从首页直接发送第一条消息，可由前端跳转到新创建的会话路由

### 4.2 `/conversations`

用于承载对话页，但未明确选中某个会话。

适合场景：
- 用户通过左侧导航进入对话页
- 用户当前处于“只有列表，没有选中详情”的状态

V1 说明：
- 如果产品希望减少 `/` 和 `/conversations` 的重复，后续也可以考虑把首页空态直接合并到 `/conversations`
- 但在当前阶段，保留 `/` 作为更清晰的首页入口会更稳

### 4.3 `/conversations/:conversationId`

用于打开某个具体会话。

页面表现：
- 左侧栏展示文件夹和对话列表
- 主区展示当前会话内容
- 底部显示输入框

这是 V1 最核心的详情路由。

## 5. 文件夹相关参数态

文件夹更适合作为参数或查询态，而不是单独做顶层路由。

建议方式：
- `/conversations?folder=all`
- `/conversations?folder=:folderId`

V1 说明：
- 这样可以保留“当前浏览的是哪个文件夹”的可分享和可恢复状态
- 但仍然保持 `conversations` 是主路由

## 6. Channels 路由

### 7.1 `/channels`

用于展示 Channel 列表页或空态。

页面表现：
- 左侧栏显示 Channel 列表
- 主区显示空态或默认引导

### 7.2 `/channels/:channelId`

用于打开某个 Channel 详情。

页面表现：
- 左侧栏显示 Channel 列表
- 主区显示当前 Channel 的连接状态、最近消息和相关对话

## 7. 任务路由

### 8.1 `/tasks`

用于展示任务列表页或默认空态。

页面表现：
- 左侧栏显示任务列表和筛选
- 主区显示空态或创建引导

### 8.2 `/tasks/:taskId`

用于打开某个任务详情。

页面表现：
- 左侧栏显示任务列表
- 主区显示任务详情、下次执行时间、执行记录和结果

### 7.3 任务筛选参数态

建议方式：
- `/tasks?status=active`
- `/tasks?status=paused`
- `/tasks?mode=automation`

V1 说明：
- 筛选适合进 URL
- 创建任务的弹层或表单过程不需要单独路由

## 8. Skills 路由

### 9.1 `/skills`

用于展示 Skills 列表页或默认空态。

### 9.2 `/skills/:skillId`

用于打开某个 Skill 详情。

页面表现：
- 左侧栏显示 Skills 列表
- 主区显示 Skill 简介、说明和启用状态

### 8.3 Skills 筛选参数态

建议方式：
- `/skills?status=enabled`
- `/skills?source=built_in`

## 9. 设置路由

### 10.1 `/settings`

用于展示低频系统设置页。

建议先保持单一路由，不拆子路由。

页面内容可包含：
- 权限模式
- 默认模型
- 默认推理强度
- 连接状态相关配置
- 关于信息

V1 说明：
- 由于设置页较轻，先不做 `/settings/permissions` 之类的子路由

## 10. 不建议进入路由的状态

以下状态建议保留在本地 UI 状态中，不进入 URL：

- 输入框草稿内容
- 上传中的文件列表
- 推理强度下拉是否展开
- 模型选择器是否展开
- 文件夹新建或重命名弹窗
- 删除确认弹窗
- Channel 连接向导中的临时步骤
- 任务创建表单中的未提交内容

## 11. 推荐的页面骨架关系

V1 建议所有主路由共用同一个 `AppShell`：

- 左侧栏固定存在
- 主区根据路由切换内容
- 输入框默认只在首页和对话页出现
- 设置页作为低频页，仍复用同一壳层

这样做的好处：
- 体验统一
- 状态更容易保留
- 和当前 UI 方向一致

## 12. 推荐的初版路由树

```text
/
/conversations
/conversations/:conversationId
/channels
/channels/:channelId
/tasks
/tasks/:taskId
/skills
/skills/:skillId
/settings
```

## 13. V1 需要守住的路由约束

- 一级入口和主路由尽量保持一一对应
- 详情页使用路径参数，筛选使用查询参数
- 不把所有交互都路由化
- 首页保持极简，不额外挂很多首页专属子路由
- 设置页先保持单一路由

## 14. 后续可以继续细化的部分

在这份草案基础上，下一步最适合继续补的是：
- 页面级加载策略
- 路由和数据预取关系
- 路由切换时的状态保留策略
- 前端工程初始化方案
